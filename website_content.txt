It is inspired by AIOpsLab — Microsoft Research's open benchmark framework for AI-powered IT operations. AIOpsLab defined the problem space: how do you evaluate AI agents against realistic production failures? ACE answers it with chaos engineering, statistical rigour, and a formal certification report. The chaos engineering layer is built on LitmusChaos — a Linux Foundation Networking (LFN) project — extended with agentic scenario support: trace-correlated fault injection, agent health scheduling, and the statistical certification pipeline that turns raw Langfuse traces into a formal report.    We put the SRE agent through 30 failure scenarios. Here is what we found.  The agent under test is an SRE agent built on top of Microsoft's Flash ITOps agent — an LLM-powered system capable of investigating alerts, pulling Kubernetes traces, running diagnostics, and executing runbooks autonomously via MCP. The target environment is SockShop, the same microservices benchmark used by AIOpsLab and Microsoft Research. The question we needed to answer was not "does it work?" — it had already passed internal testing. The question was what happens when production is genuinely broken at the infrastructure level: when the monitoring service the agent depends on is itself degraded, when pod-status tool calls return stale data, when the runbook vector store times out mid-investigation, when disk-fill and container-kill arrive simultaneously at 3 am.   30+ chaos scenariosinjected   30× independent runsper scenario   12 report sectionsgenerated   P95 time-to-detectmeasured   The result was a 12-section certification report with real numbers across detection rate, P95 time-to-detect, tool-call failure handling, hallucination rate, and PII exposure under stress. Some results were reassuring. Some told us exactly what to fix before shipping. All of it was documented, reproducible, and shareable with anyone who needed to understand whether this agent was ready. That is the report AgentCert generates. The SRE agent is the first thing it certified.    Same architecture. Every domain.  The three phases that worked for the SRE agent — fault injection, systematic measurement, formal report — apply to any agent in any domain. What changes between domains is the fault library: the catalogue of failure scenarios that matter for your stack and your risk profile. ACE ships with the SRE library and a framework for extending it. The community builds the rest.   🏥 Healthcare &amp; Life Sciences Diagnostic triage · clinical decision support · adverse-event detection When a records service degrades, does the agent fail safely or continue on incomplete data? ACE is an open source project extending Microsoft Research's AIOpsLab with chaos engineering, statistical certification, and a formal report standard. The most important thing it can become is the open benchmark for AI agent reliability across every domain — and that requires contributors from every domain.   Extend AIOpsLab fault scenarios AIOpsLab defines the benchmark. ACE makes it certifiable. Add fault scenarios for your cloud provider, agent framework, or domain — growing the shared library that every certification draws from.   Certify your own agent Have an ITOps, SRE, or domain agent built on AutoGen, LangChain, Semantic Kernel, or your own stack? Run ACE against it. The certification framework is agent-agnostic by design.   Gate deployments on certification The ACE API integrates with CI/CD. Run a certification experiment as a deployment gate — if detection rate drops below threshold, the deploy doesn't go. Reliability as a hard requirement, not a soft hope.   Build domain certifiers The SRE certifier is the first. Healthcare, finance, legal, and DevSec each have different failure modes and different evidence requirements. Build the certifier for your domain and contribute it as a plugin.
